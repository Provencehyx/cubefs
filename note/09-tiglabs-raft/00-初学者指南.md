# tiglabs/raft 初学者指南

> 本文档自上而下讲解 tiglabs/raft 的架构，适合初学者阅读。

## 一、你需要先了解的背景

### 1.1 什么是 Raft？

Raft 是一个**分布式共识协议**，用于让多台机器就"数据应该是什么"达成一致。

**生活类比**：想象一个 3 人小组需要做决定：
- 必须选出一个**组长（Leader）**
- 任何决定必须**组长提出**，并获得**至少 2 人同意**
- 如果组长失联，剩下的人**重新选举**新组长

### 1.2 为什么需要 Multi-Raft？

CubeFS 的 MetaNode 管理着成千上万个 MetaPartition，如果每个 Partition 都独立运行一个 Raft 进程：
- 资源浪费：每个 Raft 都需要定时器、网络连接
- 管理困难：进程数量爆炸

**Multi-Raft 解决方案**：
```
┌─────────────────────────────────────────────────────────────┐
│                     一个进程内                               │
│                                                             │
│   共享资源:                                                 │
│   - 1 个定时器（而不是 N 个）                               │
│   - 1 套网络连接（而不是 N 套）                             │
│                                                             │
│   独立资源（每个 Partition 独有）:                          │
│   - Raft 状态机                                             │
│   - 日志存储                                                │
│   - 业务状态机                                              │
│                                                             │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐                      │
│   │ Raft 1  │ │ Raft 2  │ │ Raft 3  │ ...                  │
│   │ (MP-1)  │ │ (MP-2)  │ │ (MP-3)  │                      │
│   └─────────┘ └─────────┘ └─────────┘                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 三、核心概念详解

### 3.1 三种角色

| 角色 | 职责 | 数量 |
|------|------|------|
| **Leader** | 处理所有客户端请求，复制日志给 Follower | 最多 1 个 |
| **Follower** | 被动接收 Leader 的心跳和日志，投票 | N-1 个 |
| **Candidate** | 选举过程中的临时状态 | 选举期间 |

### 3.2 关键术语

| 术语 | 含义 | 示例 |
|------|------|------|
| **Term（任期）** | 逻辑时钟，每次选举 +1 | Term=5 表示第 5 任 Leader |
| **Index（索引）** | 日志的序号，从 1 开始递增 | Index=100 是第 100 条日志 |
| **Committed** | 已被多数节点确认的日志 | 3 节点中 2 个确认 |
| **Applied** | 已应用到状态机的日志 | applied <= committed |

### 3.3 状态转换

```
                     启动
                       │
                       ▼
                ┌─────────────┐
                │  Follower   │◄─────────────────────────┐
                └──────┬──────┘                          │
                       │                                 │
              选举超时 │                                 │ 发现更高 term
              (无心跳)  │                                 │
                       ▼                                 │
                ┌─────────────┐                          │
                │  Candidate  │──────────────────────────┤
                └──────┬──────┘     选举失败/超时        │
                       │                                 │
              获得多数票│                                 │
                       ▼                                 │
                ┌─────────────┐                          │
                │   Leader    │──────────────────────────┘
                └─────────────┘     发现更高 term
```

## 四、数据流：一条命令是如何被处理的？

### 4.1 简化流程

```
客户端请求: "创建文件 /a/b/c"
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│ 1. MetaPartition.Submit(cmd)                                │
│    业务层将命令提交给 Raft                                   │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. raft.propose(cmd)                                        │
│    检查是否是 Leader，构造 Entry，放入 propc 通道            │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. raftFsm.Step(LocalMsgProp)                               │
│    - Leader 追加日志到 raftLog                               │
│    - 生成 MsgApp 消息                                        │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. Transport.Send(MsgApp)                                   │
│    发送日志追加请求给所有 Follower                           │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. Follower 处理 MsgApp                                     │
│    - 追加日志到本地                                          │
│    - 返回 MsgAppResp                                        │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 6. Leader 收到多数 MsgAppResp                               │
│    - 更新 committed 索引                                     │
│    - 生成 apply 任务                                         │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 7. runApply() 处理                                          │
│    - 调用 StateMachine.Apply(cmd)                           │
│    - 通知客户端结果                                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 关键代码路径

```go
// 入口：业务层调用
RaftServer.Submit(id, cmd)
    │
    ├──► raft.propose(cmd, future)           // raft.go
    │        │
    │        └──► propc <- proposal          // 发送到通道
    │
    └──► run() 主循环处理                     // raft.go:273
             │
             ├──► case pr := <-propc:
             │        raftFsm.Step(LocalMsgProp)
             │
             └──► handleReady()
                      │
                      ├──► persist()          // 持久化
                      ├──► apply()            // 生成 apply
                      └──► sendMessage()      // 发送消息
```

## 五、重点设计解析

### 5.1 为什么 raft 实例有两个 goroutine？

```
┌─────────────────────────────────────────────────────────────┐
│  问题：如果只有一个 goroutine                               │
│                                                             │
│  run() {                                                    │
│      处理消息 → 更新状态 → 应用日志(慢!) → 处理下一条消息    │
│  }                                                          │
│                                                             │
│  结果：应用日志可能涉及磁盘写入、复杂计算，阻塞协议处理       │
│       → 心跳超时 → 不必要的选举                              │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  解决：分离协议处理和日志应用                                │
│                                                             │
│  run()                    runApply()                        │
│  ┌──────────────────┐    ┌──────────────────┐               │
│  │ 处理消息          │    │ 调用 Apply()     │               │
│  │ 更新状态          │    │ (可以慢)         │               │
│  │ 发送心跳          │    │                  │               │
│  │ (必须快)          │    │                  │               │
│  └────────┬─────────┘    └────────▲─────────┘               │
│           │                       │                         │
│           │      applyc           │                         │
│           └───────────────────────┘                         │
│                                                             │
│  好处：心跳/选举不受 Apply 速度影响                          │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 为什么用两个网络端口（心跳 + 复制）？

```
┌─────────────────────────────────────────────────────────────┐
│  问题：心跳和大日志包共用连接                                │
│                                                             │
│  ┌────────────────────────────────────────────┐            │
│  │  大日志包 (MB级)                            │            │
│  │  ████████████████████████████████████████  │ ← 正在发送  │
│  │  心跳包 (几十字节)                          │            │
│  │  ░░░░                                      │ ← 被阻塞！  │
│  └────────────────────────────────────────────┘            │
│                                                             │
│  结果：心跳延迟 → Follower 认为 Leader 挂了 → 发起选举      │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  解决：双端口设计                                            │
│                                                             │
│  HeartbeatPort (17230)         ReplicaPort (17240)         │
│  ┌──────────────────┐          ┌──────────────────┐        │
│  │ 心跳 ░░░░░░░░     │          │ 大包 ████████████ │        │
│  │ 投票请求         │          │ 日志追加         │        │
│  │ (小、快、优先)   │          │ 快照             │        │
│  └──────────────────┘          └──────────────────┘        │
│                                                             │
│  好处：心跳永远不会被大包阻塞                                │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 tick 机制：共享定时器

```
┌─────────────────────────────────────────────────────────────┐
│  传统方案：每个 Raft 独立定时器                              │
│                                                             │
│  raft1: time.NewTicker() → goroutine 1                     │
│  raft2: time.NewTicker() → goroutine 2                     │
│  raft3: time.NewTicker() → goroutine 3                     │
│  ...                                                        │
│  raft1000: time.NewTicker() → goroutine 1000               │
│                                                             │
│  问题：1000 个 Raft = 1000 个定时器 + 1000 个 goroutine     │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  tiglabs 方案：共享定时器                                    │
│                                                             │
│  RaftServer: 1 个 ticker                                    │
│      │                                                      │
│      │  case <-ticker.C:                                    │
│      │      for _, raft := range rafts {                    │
│      │          raft.tick()  // 通知所有 raft               │
│      │      }                                               │
│      │                                                      │
│  好处：                                                     │
│  - 资源开销恒定，与 Raft 数量无关                           │
│  - tick 同步，避免选举风暴                                  │
└─────────────────────────────────────────────────────────────┘
```

## 六、核心结构体速查

### 6.1 RaftServer

```go
// 位置: server.go:31
type RaftServer struct {
    config *Config               // 全局配置
    ticker *time.Ticker          // 共享定时器
    heartc chan *proto.Message   // 心跳消息通道
    rafts  map[uint64]*raft      // ★ 所有 Raft Group
}
```

**要点**：
- `rafts` 是一个 map，key 是 Raft Group ID（对应 MetaPartition ID）
- `ticker` 是共享的，驱动所有 Raft Group 的 tick

### 6.2 raft

```go
// 位置: raft.go:99
type raft struct {
    raftFsm    *raftFsm          // Raft 协议状态机
    curApplied util.AtomicUInt64 // 当前已应用索引
    pending    map[uint64]*Future // 等待响应的提案
    
    // 核心通道
    propc     chan *proposal      // 提案输入
    applyc    chan *apply         // 应用输出
    recvc     chan *proto.Message // 网络消息
    tickc     chan struct{}       // tick 信号
}
```

**要点**：
- 每个 raft 有自己的通道，独立处理
- `pending` 跟踪等待响应的客户端

### 6.3 raftFsm

```go
// 位置: raft_fsm.go:49
type raftFsm struct {
    id       uint64          // Raft Group ID
    term     uint64          // 当前任期
    vote     uint64          // 投票给谁
    leader   uint64          // 当前 Leader
    state    fsmState        // Leader/Follower/Candidate
    raftLog  *raftLog        // 日志管理
    replicas map[uint64]*replica // 副本进度
    
    step stepFunc            // ★ 状态处理函数（多态）
    tick func()              // ★ tick 处理函数（多态）
}
```

**要点**：
- `step` 和 `tick` 是函数指针，根据当前状态指向不同函数
- 这是一种**状态模式**的实现

### 6.4 raftLog

```go
// 位置: raft_log.go:31
type raftLog struct {
    unstable  unstable        // 内存中未持久化的日志
    storage   storage.Storage // 已持久化的日志
    committed uint64          // 已提交索引
    applied   uint64          // 已应用索引
}
```

**要点**：
- 日志分两部分：`unstable`（内存）和 `storage`（磁盘）
- 关系：`applied <= committed <= lastIndex`

| 消息 | 方向 | 用途 | 频率 |
|------|------|------|------|
| `LocalMsgHup` | 本地 | 触发选举 | 低 |
| `LocalMsgProp` | 本地 | 客户端提案 | 高 |

| `RespMsgHeartBeat` | F→L | 心跳响应 | 高 |
| `ReqMsgSnapShot` | L→F | 快照传输 | 低 |

## 八、常见问题

### Q1: 为什么 Follower 收不到心跳会触发选举？

**答**：这是 Raft 的核心安全机制。如果 Leader 真的挂了，Follower 必须能够检测到并选出新 Leader，否则整个集群会停止服务。

### Q2: 为什么要先复制后提交？

**答**：确保数据安全。只有当多数节点都有这条日志后，才认为它是"committed"，即使 Leader 挂了，新 Leader 也一定有这条日志。

### Q3: 什么时候会触发快照？

**答**：当日志太多时。想象日志有 100 万条，新节点加入需要从头复制，太慢了。快照是当前状态的"压缩版"，新节点只需要接收快照 + 之后的少量日志。

### Q4: PreVote 是什么？

**答**：防止网络分区时的"选举风暴"。节点在真正发起选举前，先发一轮 PreVote 探测自己是否有可能赢，避免无谓地增加 term。
